# Routing

Smart routing lets a virtual API key define candidate models by model type. The organisation must enable routing, and the API key must have a routing policy.

## Model Types

Routing policies are keyed by model type, for example `chat` or `embeddings`. Odock resolves the requested model type from the configured model record.

## Failover Triggers

Default failover triggers are `5xx` and `timeout`. You can also configure `rate_limit` or `any`. Use `any` carefully because it can hide application errors by retrying too broadly.

## Routing Strategy

Strategies control how candidates are ordered.

### Next Candidate

On a failover-triggering error, Odock tries the next candidate until the request succeeds or the retry cap is reached.

### Priority

Priority sorts candidates by ascending priority. Lower numbers are tried first. This is useful for primary/secondary provider plans.

### Round Robin

Round robin rotates the first candidate per API key. Use it for simple traffic distribution across equivalent models.

## Candidates

Candidates are model names configured in the same organisation and accessible to the API key.

### Multi Model Candidates

Unified endpoint candidates can span providers as long as each candidate has the same model type as the requested model.

### Native Provider Fallbacks

Native endpoints pin the provider family. Native fallback policies map provider names, such as `openai`, `anthropic`, `gemini`, or `vllm`, to model names that should be tried after the requested model.

# Intro

Odock is an AI governance platform for LLM and MCP traffic. It gives teams one controlled entry point for model calls, provider credentials, model access, MCP tool access, routing, budgets, quotas, usage records, security checks, plugins, and observability.

The system has two main parts:

- `odock-server`: the Go gateway that receives runtime traffic and enforces authentication, access, routing, safety, policy, budget, quota, usage, and observability controls.
- `odock-ui`: the Next.js management UI used to configure organisations, users, teams, providers, models, MCP servers, virtual API keys, budgets, quotas, and usage views.

For practical gateway examples, see `odock-server/tests/_python`. Those scripts show OpenAI-style calls, Anthropic native calls, Gemini calls, vLLM calls, unified multi-model calls, and MCP validation flows.

## Architecture

Odock separates configuration from traffic execution. Users configure governance in the UI; the gateway reads those records from Postgres, caches hot metadata in Redis, and applies the controls on each request.

The LLM request path is:

1. Receive a gateway request.
2. Apply request ID, logging, metrics, concurrency, and pre-auth rate checks.
3. Authenticate the virtual API key.
4. Resolve organisation, team, user, API key, model, provider, and provider key context.
5. Run policy checks, model access checks, budgets, quotas, SafetySec, and plugins.
6. Call the selected upstream provider.
7. Transform the response into the endpoint shape requested by the client.
8. Record usage, cost, routing attempts, latency, status, and provider usage.
9. Run post-response SafetySec and plugin hooks.

The MCP request path is similar, but it resolves an MCP server by id or slug, verifies `ApiKeyMcpAccess`, applies MCP tool and semantic rules, proxies to `STREAMABLE_HTTP`, `SSE`, or `STDIO`, and records MCP-specific usage artifacts.

## Install

For local development, run each project from its own directory.

Backend:

```bash
cd odock-server
POSTGRES_DSN='postgres://odock:odock@localhost:5432/odock?sslmode=disable' go run ./cmd/gateway
```

Frontend:

```bash
cd odock-ui
yarn install
npx prisma migrate dev
yarn dev
```

The gateway defaults to `0.0.0.0:8080`. Redis defaults to `localhost:6379` and is used for caching, rate limiting, routing policy cache, SafetySec sessions, and related hot-path state.

## Quick Start

1. Create or select an organisation in the UI.
2. Add a provider, for example OpenAI, Anthropic, Google, vLLM, Azure OpenAI, or Custom.
3. Add an encrypted provider API key for that provider.
4. Add one or more models and connect each model to a provider and provider key.
5. Create a virtual API key scoped to the organisation, team, or user.
6. Grant the virtual API key access to the models it can use.
7. Call the gateway with the virtual API key.

Minimal OpenAI SDK example:

```python
from openai import OpenAI

client = OpenAI(
    api_key="sk_your_dock_virtual_key",
    base_url="http://localhost:8080/v1",
)

response = client.chat.completions.create(
    model="gpt-4.1-mini",
    messages=[{"role": "user", "content": "Hello through Odock"}],
)

print(response.choices[0].message.content)
```
